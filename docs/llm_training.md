# LLM Training With Coordinate Tokens

The compression result is useful for LLMs when it reduces the number of tokens
used during training.

The proposed workflow is:

```text
text corpus
  -> build shared coordinate universe
  -> encode corpus as coordinate-token sequences
  -> train language model on coordinate tokens
  -> generate coordinate tokens
  -> decode generated tokens back to text
```

The key comparison is not only compressed file size. It is:

```text
baseline training token count
vs
coordinate training token count
```

If the coordinate stream is shorter and learnable, the model can process the
same source corpus with fewer sequence positions. This could reduce training
steps per document, improve effective context density, or increase the amount of
source information that fits into a fixed context window.

The codec must prove four things before model training is meaningful:

| Requirement | Why it matters |
|---|---|
| exact roundtrip | training data can be reconstructed exactly |
| token reduction | coordinate corpus has fewer target symbols |
| stable universe | encoder and decoder share deterministic meanings |
| measurable payload | compression can be compared against standard methods |

After that, the model experiment should measure:

| Metric | Question |
|---|---|
| validation loss on coordinate tokens | can the model learn the compressed language? |
| decoded text quality | does generated coordinate output decode into useful text? |
| generated bytes per token | does each model token carry more decoded text? |
| training throughput | do fewer tokens reduce end-to-end training time? |
| context density | does a fixed context window represent more source text? |

V1 provides the codec and token-count measurement layer. Model architecture and
training choices can be tested on top of this representation.
